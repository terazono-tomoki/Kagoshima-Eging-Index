# Kagoshima-Eging-Index

鹿児島県内の漁港・ポイントに絞ったエギング専用のコンディション判定ツールです。

## 使用している外部 API（データソース）

本アプリは API キー不要 の [Open-Meteo](https://open-meteo.com/) を HTTP で取得しています（`urllib.request`）。

| エンドポイント | 役割 | 取得する変数（hourly） |
|----------------|------|-------------------------|
| `https://api.open-meteo.com/v1/forecast` | 一般気象の予報 | `wind_speed_10m`, `pressure_msl` |
| `https://marine-api.open-meteo.com/v1/marine` | 海洋の予報 | `wave_height`, `sea_surface_temperature`, `sea_level_height_msl` |

### リクエストの共通パラメータ

- 座標: 各地点の緯度・経度（アプリ内 `locations` に固定）
- タイムゾーン: `Asia/Tokyo`
- 週間予報: `forecast_days=7` を指定し、時系列を日ごとに集約
- 単日スナップショット（釣果ログ登録時の気象記録用）: `start_date` / `end_date` を同一日に指定

取得結果は気象・海洋の hourly を時刻で結合し、欠損行を除いたうえで日別に集約します。  
風・波・水温・気圧は同日の平均、潮は同日の hourly `sea_level_height_msl` の最大値と最小値の差（その日の潮差の目安）からスコア化します。キャッシュは `st.cache_data(ttl=3600)` で約1時間です。

`sea_level_height_msl` は Open-Meteo が提供する 潮汐を含む海面高度のグリッドモデルです（無料・APIキー不要）。気象庁の潮汐表や港湾ごとの天文潮とは基準・精度が異なり、航海用には推奨されません。本アプリでは「その地点付近でその日の潮の変動が大きいか」の目安に使っています。

## 独自ロジックの概要

「エギング向けの釣りやすさ」は 実験的なスコアリング であり、実海況・潮流・地形は反映していません。参考用途としてください。

### 1. 潮（海面高度モデルからの日次潮差）

**何を見ているか**  
Open-Meteo が返す「その地点の海面の高さ（潮汐を含むモデル値）」を、**1時間ごと**に並べたうえで、**同じ日に属するデータだけ**を対象にします。

**潮差の求め方（その日の「動き」の大きさ）**  
その日24時間ぶんの海面高度のうち、**いちばん高い値といちばん低い値の差（メートル）**を「その日の潮差」とします。  
潮差が大きいほど、海面がその日のうちに大きく上下している＝**大潮に近い動き**だとみなして点を上げます。

**潮としての点数（約45点〜100点）**  
まず「基準になる潮差」を **1.85 m** と決め、実際の潮差をそれで割って **0〜1 の比率**にします（1.85 m を超える場合は比率 1 として頭打ち）。  
その比率を使って、潮だけの点数を **最低でも約45点、最高100点** に変換します（比率がゼロなら約45点、比率が最大なら100点）。

**画面に出る潮の区分（大潮寄りなど）**  
上の比率が次のどれに当てはまるかで文言を付けています。

| 比率の目安 | 表示 |
|------------|------|
| 0.82 以上 | 大潮寄り |
| 0.62 以上（かつ 0.82 未満） | 中潮寄り |
| 0.42 以上（かつ 0.62 未満） | 小潮寄り |
| 0.42 未満 | 長潮/若潮寄り |

**総合スコアとの関係**  
この「潮の点数」は、風・波・水温・気圧の点数とあわせて **全体の 34% の割合**で混ぜ込んでいます（残りは風26%、波18%、水温14%、気圧8%）。

**注意**  
気象庁の潮汐表そのものではなく、グリッド上のモデル値です。海岸や地形の影響で、現地の実際の潮とずれることがあります。

### 2. 総合スコア（Open-Meteo の日次平均から）

風・波・水温・気圧は、その日の **平均値** を使います。それぞれ「釣り向きらしさ」を **0〜100 くらいの点数** に直したうえで、潮と **割合を決めて混ぜたもの**が総合スコア（0〜100）です（`evaluate_eging_condition`）。

**なぜ「係数」を使うか**  
風は m/s、波は m、水温は℃、気圧は hPa と **単位も変化の出方も違う**ので、そのまま足し合わせられません。そこで「弱い風ほどよい」「波は低いほどよい」のようにルールを決め、**実測値から点数へ変換するときに倍率（係数）を掛ける**形にしています。  
例として、風は「100 − 風速×9.5」、波は「100 − 波高×38」、水温は「20℃を中心に外れるほど減点（差×8.5）」、気圧は「1016 hPa を目安に外れるほど減点（差×3.5）」です。  
これらはデータから自動で求めた最適値ではなく、**だいたい同じ物差しにそろえるための目安**です。

| 要素 | スコアの考え方（概要） | 総合への重み |
|------|-------------------------|--------------|
| 潮 | 日次潮差（海面高度モデル）からの潮スコア | 34% |
| 風 | 弱いほど高評価（風速 × 係数で減点） | 26% |
| 波 | 低いほど高評価（波高 × 係数で減点） | 18% |
| 水温 | 20℃ 付近を最高とし、乖離で減点 | 14% |
| 気圧 | 1016 hPa 付近を目安に安定寄りを評価 | 8% |

総合スコアが次のどの区間に入るかで、**S〜D のランク**を付けます（`get_rank`）。

| ランク | 条件（総合スコア） |
|--------|---------------------|
| S | 85 以上 |
| A | 73 以上 85 未満 |
| B | 60 以上 73 未満 |
| C | 48 以上 60 未満 |
| D | 48 未満 |

### 3. 釣果ログ連動（任意）

同じポイントのログが **2件に達しない** と、「過去の釣果と今日が似ているか」は判定せず **データ不足** とだけ表示します（`evaluate_from_catch_records`）。

**やっていることの流れ**

1. いま選んでいるポイント名と同じ場所のログだけを集めます。
2. **ログ1件ごとに**、「今日の風・波・水温・気圧」と「そのログを取った日の同じ4つ」を比べ、ばらつきを **ひとつの数字（距離）** にまとめます。  
   計算は「それぞれの差の絶対値×重み」を足すだけです：  
   **風の差×1.2 ＋ 波の差×8.0 ＋ 水温の差×1.1 ＋ 気圧の差×0.15**  
   （単位がばらばらなので、数字の大きさを寄せるための重みです。）
3. その距離が **小さい**ほど、そのログの日のコンディションに今日は近い、という意味になります。距離をそのまま使うのではなく、**「100から距離×4.2を引いたもの」**をそのログの **類似スコア** にしています（マイナスにならないよう下限は0）。だから **距離が小さいほど類似スコアは高く**なります。
4. 対象ログが複数あるときは、各ログの類似スコアを **単純平均**します。**平均が高いほど**、過去に釣れた日の条件に今日が近い、と言っています（「距離だけ」を平均しているわけではありません）。

**「実績一致：高／中／低」の付け方**  
上の平均スコアが次のどれに入るかで決めます。

| 画面の表示 | 平均スコアの目安 |
|------------|------------------|
| 実績一致: 高 | 70 以上 |
| 実績一致: 中 | 52 以上 70 未満 |
| 実績一致: 低 | 52 未満 |

重み・`4.2`・閾値の数字は、機械学習で最適化したものではなく **決め打ちの目安**です。

### 4. ローカル保存・認証

- 釣果は `catch_records.json`、写真は `catch_images/` に保存
- 記録欄の閲覧・保存は `.streamlit/secrets.toml` の `records_section_password` で制限可能
